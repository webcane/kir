**Reviewed:** 2026-07-01

## Summary

Two files were reviewed: the `DocumentCompiler` service that wires the four document passes into a pipeline and persists to a repository, and the integration tests covering STOR-01 and STOR-02. The wiring itself is minimal and structurally correct — the pipeline validates at construction, passes run in dependency order, and the final artifact is saved. One correctness defect was found: no guard exists to prevent persisting a Document with an empty `id` if the metadata pass is missing or fails, and no error handling surrounds the `save()` call, meaning persistence failures are indistinguishable from pipeline failures to callers. The test file has a dead construction (lines 168) and violates the explicit project ban on `from __future__ import annotations`.

## Critical Issues

### CR-01: No guard against persisting an empty-id artifact; `save()` exceptions propagate unhandled

**File:** `src/kir/compiler/documents/compiler.py:84`

**Issue:** `self._ctx.repository.save(ir.id, ir.model_dump())` is called unconditionally after the pipeline loop with no validation and no error handling. Two sub-problems compound here:

1. The initial `Document` is constructed with `id=""` at line 71. The `metadata_pass` populates `id` via `_slugify()`, and the topological sort guarantees it runs before `extract_concepts_pass`. However, if `metadata_pass` itself raises an unhandled exception, the pipeline loop re-raises that exception and `save()` is never called — which avoids the empty-id store. But if the pipeline is ever restructured, or if `metadata_pass` is not registered (the registry accepts any set of passes), the call to `save("", ...)` proceeds unchecked. The `YamlFileRepository` validates `artifact_id` against an allowlist and would raise `ValueError` on an empty string; an `InMemoryFakeRepository` silently stores it under key `""`. In the test environment this means a corrupt entry lands in the store without error.

2. `repository.save()` can raise `OSError`, `PermissionError`, or repository-specific validation errors. These bubble through `compile()` uncaught. The caller receives an exception after all passes (including LLM calls) have already succeeded, with no structured Diagnostic and no returned IR. The caller cannot distinguish "pipeline failed" from "persistence failed" without catching concrete implementation types, breaking the port abstraction.

**Fix:** Add a pre-save guard and wrap the save call:

```python
        if not ir.id:
            raise RuntimeError(
                "DocumentCompiler: pipeline completed but Document.id is empty. "
                "Ensure metadata_pass is registered and ran successfully."
            )
        try:
            self._ctx.repository.save(ir.id, ir.model_dump())
        except Exception as exc:
            raise RuntimeError(
                f"DocumentCompiler: failed to persist document {ir.id!r}: {exc}"
            ) from exc
        return ir
```

**Issue:** The style guide (`STYLE_GUIDE.md`, "Required: no `__future__` annotations") and the project memory notes explicitly forbid `from __future__ import annotations` across all files. ### WR-02: Dead compiler construction — `_make_compiler_context()` result is immediately discarded

**File:** `tests/compiler/documents/test_document_compiler.py:168`

**Issue:** `test_document_compiler_pipeline_has_four_passes` calls `_make_compiler_context()` at line 168, assigns both return values to `_`, and discards them. The test then builds a second, nearly identical `CompilerContext` and `DocumentCompiler` from scratch on lines 170–183 for the actual assertion. The first call is dead code: it allocates a `FakeLLMAdapter`, `InMemoryFakeRepository`, full `CompilerContext`, and `DocumentCompiler` that are never used. The comment "build compiler via helper then inspect" is wrong — the helper's result is not inspected. This misleads the reader into thinking the helper and the second construction differ in a meaningful way.

**Fix:** Delete line 168 entirely. The remaining direct construction is sufficient.

### WR-03: STOR-02 test uses separate repositories — structurally cannot detect same-repository id collision

**File:** `tests/compiler/documents/test_document_compiler.py:231-306`

**Issue:** `test_two_documents_produce_two_distinct_repository_entries` creates `repo_a` and `repo_b` as two independent `InMemoryFakeRepository` instances, writes one document into each, and asserts `key_a != key_b`. This assertion is trivially true: two independent stores with one entry each will always have unrelated in-memory identity for their keys. The test cannot detect the STOR-02 failure it is supposed to guard — namely that compiling two documents with identical titles (both slugify to the same id, e.g., `"untitled"`) into the **same** repository causes a silent overwrite. Using separate repositories removes the collision surface entirely.

The test as written gives false confidence. A bug where `DocumentCompiler` always writes to key `"untitled"` regardless of content would still pass this test.

**Fix:** Compile both documents into a **single shared repository** and assert that two distinct keys are present:

```python
repo = InMemoryFakeRepository()
ctx_a = CompilerContext(llm=adapter_a, repository=repo, ...)
ctx_b = CompilerContext(llm=adapter_b, repository=repo, ...)
compiler_a = DocumentCompiler(document_registry, ctx_a)
compiler_b = DocumentCompiler(document_registry, ctx_b)

await compiler_a.compile(doc_a_path)
await compiler_b.compile(doc_b_path)

assert len(repo._store) == 2, "Expected 2 distinct entries; id collision may have occurred"
key_a, key_b = list(repo._store.keys())
assert key_a != key_b
```

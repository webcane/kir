**Reviewed:** 2026-07-01

## Summary

This phase wires `DocumentCompiler.compile()` to call `self._ctx.repository.save(ir.id, ir.model_dump())` at the end of compilation, and adds two new async tests (STOR-01, STOR-02) asserting persistence behavior. The implementation is minimal — one line added to `compiler.py`. The problems are concentrated in correctness gaps and test design failures rather than the wiring itself.

Two critical issues were found: (1) `repository.save()` exceptions are not handled, meaning an I/O or validation failure at the save step leaks an uncaught exception to the caller despite the compile having already succeeded, and (2) the STOR-02 test is structurally incapable of detecting the key-collision scenario it claims to guard against, because it uses two separate repositories rather than one shared one. Three warnings cover missing error propagation contract documentation, a dead compiler construction at test line 168, and a prohibited `from __future__ import annotations` import in both test and fake files.

## Critical Issues

### CR-01: `repository.save()` exceptions propagate uncaught, breaking the compile contract

**File:** `src/kir/compiler/documents/compiler.py:84`

**Issue:** `self._ctx.repository.save(ir.id, ir.model_dump())` is called with no exception handling. The `YamlFileRepository.save()` can raise `ValueError` (when `artifact_id` fails its allowlist pattern) and any I/O exception (`OSError`, `PermissionError`) from `open()` or `ruamel.yaml.dump()`. If the repository raises, the caller never receives the compiled `Document` even though all passes completed successfully. This violates the compiler's implicit contract that a successful pipeline produces a returned IR — the caller has no way to distinguish "pipeline failed" from "persistence failed."

The `ir.id` is produced by `_slugify()` which enforces `[a-z0-9-]+` — characters also accepted by `_SAFE_ARTIFACT_ID` (`[A-Za-z0-9_-]+`) — so `ValueError` is unlikely in practice, but file-system failures are not. Concretely, if the output directory is read-only, `compile()` raises `PermissionError` after all LLM calls have been made and billed.

**Fix:** Wrap the save call and propagate a domain-typed exception so the caller can distinguish persistence failure from pipeline failure:

```python
from kir.core.domain.exceptions import PersistenceError   # define if not yet exists

try:
    self._ctx.repository.save(ir.id, ir.model_dump())
except Exception as exc:
    raise PersistenceError(
        f"Failed to persist document {ir.id!r}: {exc}"
    ) from exc
return ir
```

Alternatively, if the project intends `compile()` to be fault-tolerant, append a `Diagnostic` and still return `ir` — but this must be a conscious decision, not silence.

### CR-02: STOR-02 test cannot detect the id-collision scenario it is supposed to guard

**File:** `tests/compiler/documents/test_document_compiler.py:231-306`

**Issue:** `test_two_documents_produce_two_distinct_repository_entries` creates **two separate repositories** (`repo_a`, `repo_b`) and compiles one document into each. It then asserts `len(repo_a._store) == 1` and `len(repo_b._store) == 1` and that `key_a != key_b`. These assertions will always pass — they are trivially satisfied the moment you write two documents into two independent in-memory stores.

The real risk this test should guard is: **compiling two headingless documents into the same repository silently overwrites the first** because both get `id="untitled"` from `metadata_pass._slugify()`. Using separate repos makes it impossible for this test to catch that failure. The test gives false confidence that STOR-02 is covered.

**Fix:** Compile both documents into a **single shared repository** and assert both keys are distinct and both entries are present:

```python
repo = InMemoryFakeRepository()
ctx_a = CompilerContext(repository=repo, ...)
ctx_b = CompilerContext(repository=repo, ...)
compiler_a = DocumentCompiler(document_registry, ctx_a)
compiler_b = DocumentCompiler(document_registry, ctx_b)

await compiler_a.compile(doc_a_path)
await compiler_b.compile(doc_b_path)

assert len(repo._store) == 2, "Expected 2 distinct entries; collision may have occurred"
key_a, key_b = list(repo._store.keys())
assert key_a != key_b
```

This also surfaces the latent design question: `id` is a content-derived slug, not a content-addressed hash, so two documents with the same title *will* collide in the repository regardless of content. That is a pre-existing design gap, but this test should at minimum prove the happy path (different titles → different keys) with a shared repo.

### WR-02: `repository.save()` contract (error behavior) is undocumented in `compile()` docstring

**File:** `src/kir/compiler/documents/compiler.py:48-65`

**Issue:** The `compile()` docstring describes what is returned but does not document what exceptions can be raised. After this phase, `compile()` can now raise whatever `RepositoryPort.save()` raises. The `RepositoryPort` protocol itself specifies no exception contract. Callers (e.g., a CLI or batch runner) cannot write correct error handling without knowing this. Per CLAUDE.md and the style guide, public methods must document their `Raises:` section.

**Fix:** Add a `Raises:` section to the `compile()` docstring:

```python
Raises:
    OSError: If the source file cannot be read.
    Exception: If repository.save() fails (e.g., PermissionError for
               filesystem-backed repositories). The compiled ir is
               NOT returned in this case.
```

### WR-03: `from __future__ import annotations` in test file and fake repository — prohibited by style guide

**File:** `tests/compiler/documents/test_document_compiler.py:12`
**File:** `tests/core/passes/fakes/fake_repository.py:5`

**Fix:** Remove `from __future__ import annotations` from both files. No other changes needed — neither file has forward references.

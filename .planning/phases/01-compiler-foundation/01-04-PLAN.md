<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `YamlFileRepository.save()`/`.load()` artifact_id -> filesystem path | `artifact_id` is the only externally-influenced input this phase that becomes a filesystem path component — untrusted in the sense that a future caller (a real pass output, Phase 2+) could construct it from LLM-derived or document-derived content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Tampering / Information Disclosure | `YamlFileRepository.save()`/`.load()` — `artifact_id` used directly as a filename | mitigate | Validate `artifact_id` against a restrictive allowlist pattern (`^[A-Za-z0-9_-]+$`) before constructing any `Path`, raising `ValueError` on violation — implemented in Task 2, proven by Task 2's Test 4 (path-traversal artifact_id test). This is the one concrete filesystem-write code path in Phase 1 and the only place ASVS V5 input validation has real teeth this phase. |
| T-01-09 | Tampering | `ruamel.yaml.YAML(typ="safe")` mode selection | mitigate | Explicitly use `typ="safe"` (not the default round-trip-preserving mode, which can be more permissive about constructing arbitrary Python objects from YAML tags) — `typ="safe"` restricts deserialization to plain Python types (dict/list/str/int/etc.), preventing arbitrary object construction from a malicious YAML file |
| T-01-10 | Tampering | npm/pip/uv installs (`ruamel.yaml` newly added this plan) | accept | Already vetted in `.planning/research/STACK.md` and re-confirmed in `01-RESEARCH.md`'s Standard Stack table ("Already locked in STACK.md for git-diff-friendly, round-trip-faithful YAML") — not a new/unaudited package, no blocking checkpoint required |
</threat_model>

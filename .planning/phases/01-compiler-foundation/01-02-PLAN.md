## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Domain model construction (Pydantic validation) | Any caller (test code now, real passes/adapters in Phase 2+) constructs domain models with arbitrary field values — Pydantic validation at the construction boundary is the only gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Tampering | All domain/IR models (Document, Concept, Relation, Taxonomy, Conflict, Diagnostic, ArtifactManifest, value objects) | mitigate | `model_config = ConfigDict(frozen=True, extra="forbid")` on every model — rejects unexpected/malformed fields at construction (ASVS V5 Input Validation) rather than silently accepting them; this is the single security control this plan introduces, applied uniformly |
| T-01-03 | Tampering | `tuple[...]` vs `list[...]` accumulating fields (e.g. `Document.sections`, `Concept.aliases`, `Diagnostic`-bearing fields) | mitigate | Use `tuple[...]`, never `list[...]`, for every accumulating field — a `list` field is internally mutable via `.append()` even on a `frozen=True` model, silently defeating CORE-07; enforced via code convention this plan and verified by the immutability test in Task 2 |
| T-01-04 | Information Disclosure | Domain layer leaking infrastructure concerns (filesystem paths, serialization format assumptions) into entity methods | mitigate | CORE-01 import-boundary audit test (Task 3) — automated, not just code review; catches `pathlib`/`yaml`/LLM SDK imports anywhere under `src/kir/core/domain/` |
</threat_model>
